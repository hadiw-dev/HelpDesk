# Phase 7 — Hardening

## What this phase adds

Phase 7 doesn't add product features — it hardens what Phases 1–6 already built: closes the security gaps those phases explicitly deferred, expands automated test coverage (unit + integration/API), and improves frontend resilience/accessibility. Every change here is defensive or verifying, not new business logic.

### Backend — Security

- **Security response headers** (`Api/Middleware/SecurityHeadersMiddleware.cs`): `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: no-referrer` on every response. This is a JSON API with no server-rendered HTML, so a full HTML-page CSP wasn't applicable — these three headers are the ones that matter for a JSON API's threat model.
- **HSTS**, enabled outside Development (`app.UseHsts()`) — meaningless until the app is actually served over HTTPS in a non-dev environment, so it's gated to the `else` branch of the existing `IsDevelopment()` check rather than always-on.
- **JWT secret key fail-fast validation** (`Program.cs`): the app now refuses to start if `Jwt:SecretKey` is missing or under 32 bytes, throwing a clear `InvalidOperationException` at startup instead of silently accepting a signature key weak enough to forge.
- **Failed-login activity logging** (`AuthService.LoginAsync`): all three failure paths (unknown/inactive account, locked-out account, wrong password) now write an `"LoginFailed"` entry via `IActivityLogService`, closing the gap where only successful logins were audited. An unknown/inactive account is logged **without** a user id, so the log itself never confirms an email is registered to anyone reading it later.
- **Magic-byte file signature validation** (`Application/Common/Utils/FileSignatureValidator.cs`, wired into `AttachmentService.UploadAsync`): closes the Phase 6-documented known limitation that upload validation was extension/declared-`Content-Type` only. The upload path now peeks the first 8 bytes of the stream and checks them against the real signature for `.pdf`, `.png`, `.jpg`/`.jpeg`, `.zip`, `.docx`, `.xlsx` (the latter two are OOXML, which is itself a zip container) before saving. Extensions with no reliable magic number (`.txt`) are intentionally left unchecked — there's nothing meaningful to verify them against. `IFormFile.OpenReadStream()` is always seekable, so the stream is rewound to position 0 after the peek.
- **Verified, not changed** (already correct from earlier phases, confirmed during this phase's audit): HTTPS redirection (`UseHttpsRedirection`), CORS allowlisting (exact-origin match, `AllowCredentials`), JWT validation parameters (issuer/audience/lifetime/signing key all validated), input validation (FluentValidation on every request DTO), SQL injection protection (100% EF Core LINQ — no raw SQL/string-concatenated queries anywhere in the codebase), XSS protection (a JSON API returns data as inert JSON text, never executes it; the frontend is a React SPA that escapes all rendered text by default), authorization (role/policy-based on every non-`[AllowAnonymous]` endpoint, resource-ownership checks via `TicketAccessGuard`), secure uploads (GUID-named files stored outside the web root — no static file middleware serves that directory at all), and structured logging (Serilog request logging + activity log table).

### Backend — Testing

- **New unit tests** (`HelpDesk.Tests`): `FileSignatureValidatorTests` (8 tests, pure logic — no DI), 4 new `AuthServiceTests` covering failed-login/lockout/success activity logging, 2 new `AttachmentServiceTests` covering the magic-byte check (mismatched content rejected, matching content accepted).
- **New integration/API tests** (`HelpDesk.IntegrationTests`, run against the real pipeline via `WebApplicationFactory<Program>` and the real LocalDB — no mocks):
  - `AuthFlowTests` — register, duplicate-email conflict, login (success/wrong-password/unknown-email), refresh-token (valid/invalid), profile (unauthenticated/authenticated/tampered-signature).
  - `AuthorizationTests` — 401 with no token, 403 with a token lacking the required role, 200 with the right role, for both role-gated (`Roles = "Admin"`) and policy-gated (`RequireManagerOrAdmin`) endpoints, plus a ticket-delete 403 check.
  - `SecurityTests` — the three new response headers are present, CORS echoes `Access-Control-Allow-Origin` only for the allowlisted origin (and omits it entirely for a disallowed one — that's how the browser actually enforces the block, not a server-side rejection), SQL-injection-style search payloads (`' OR '1'='1'`, `'; DROP TABLE Tickets; --`, etc.) return `200` with a normal paged result rather than a `500` or data loss, and a `<script>` tag in a ticket title round-trips as an inert JSON string with an `application/json` content type.
  - `TicketApiTests` — full HTTP-level CRUD lifecycle (create → get → update → delete/restore), 404 for a non-existent ticket, 400 for a missing-required-field payload, 403 for cross-user access and for an Employee attempting a delete, search returning the created ticket.
  - All new integration test classes share one `TestWebApplicationFactory` instance via an xUnit collection fixture (`ApiCollection`, `[Collection(ApiCollection.Name)]`) instead of each declaring its own `IClassFixture`. This was a genuine, non-obvious bug fix: `Program.cs` bootstraps Serilog into the process-wide static `Log.Logger`, which gets **frozen** the first time a host builds; a second independent `WebApplicationFactory` in the same test process throws `InvalidOperationException: The logger is already frozen` building its own host. See `docs/PITFALLS.md`.
- **Result**: 149 backend tests total (114 unit + 35 integration), all passing. See `docs/PHASE7_TESTING_REPORT.md` for the full breakdown and `docs/PHASE7_COVERAGE_REPORT.md` for line/branch coverage.

### Frontend

- **`ErrorBoundary`** (`components/ErrorBoundary.tsx`): a class-based React error boundary wrapping `<RouterProvider>` in `App.tsx`. React Router's own `errorElement` only catches errors thrown by a route's element and its descendants — it does not cover errors thrown by something mounted above the router (a provider, or the router's own initialization). Without this, such an error would previously unmount the whole app to a blank white screen; now it renders a friendly fallback matching `ErrorPage`'s styling.
- **Route-based code splitting** (`routes/lazyPages.ts`, `routes/AppRoutes.tsx`): all 18 page components are now `React.lazy(...)` instead of static imports, each shipping in its own chunk fetched only when that route is visited. `<Suspense>` boundaries (with a `PageLoadingFallback` spinner) were added inside `AppLayout` and `AuthLayout`, wrapping their `<Outlet/>`. Confirmed via `vite build`: each page now produces its own `dist/assets/<PageName>-*.js` chunk instead of one monolithic bundle. (The lazy definitions live in their own module rather than inline in `AppRoutes.tsx` — co-locating them with the non-component `router` export trips `eslint-plugin-react-refresh`'s Fast Refresh detection; see `docs/PITFALLS.md`.)
- **Accessibility pass**: added accessible names (a visually-hidden `<label htmlFor>` where a visible label would disrupt a compact filter bar, `aria-label` elsewhere) to every previously unlabeled `<input>`/`<select>` found across the app — ticket search/filter/sort controls (`TicketsPage`), user create/edit/search/role-filter fields (`UserManagementPage`), the activity-log action filter (`ActivityLogPage`), lookup name/description/order fields (`LookupManagementPage`), the attachment file picker (`AttachmentsPanel`), and the agent-assignment select (`AssignmentPanel`). Icon-only buttons (theme toggle, logout, notification bell) already had `aria-label`s from earlier phases; form pages built with React Hook Form (`CreateTicketPage`, `EditTicketPage`, `SystemSettingsPage`, all auth pages) already had proper `<label htmlFor>` pairs and needed no changes.
- Error handling and loading states were already present from earlier phases (per-page `isLoading`/`isError` states via TanStack Query, inline server-error messages via `extractErrorMessage`) — the `ErrorBoundary` and `PageLoadingFallback` additions are a layer *above* that, not a replacement for it.

If you want that last mile covered, run `npm run dev` and: (a) throw an error intentionally in a component to confirm the `ErrorBoundary` fallback renders, (b) open the Network tab and confirm each route lazy-loads its own chunk, (c) tab through the ticket filter bar with a screen reader or the browser's accessibility inspector to confirm the new labels are announced.

## Known limitations (carried forward / new)

- Refresh tokens are still stored as plaintext in the database (documented since Phase 2) — hashing them at rest remains a "before production" item, not addressed in this hardening pass since it's a data-model change, not a gap in the *behavior* Phase 7 was asked to verify.
- No rate-limiting/brute-force throttling beyond ASP.NET Identity's built-in lockout (5 failed attempts → temporary lockout) — a dedicated rate limiter (e.g. `Microsoft.AspNetCore.RateLimiting`) on the login/register endpoints specifically would be a reasonable next hardening step.
- No automated frontend tests (Vitest/Testing Library) were added — Phase 7's explicit ask was backend xUnit/Moq/integration/API tests plus frontend UX/accessibility improvements, not a frontend test suite; this remains a gap worth closing before this is called production-ready.
