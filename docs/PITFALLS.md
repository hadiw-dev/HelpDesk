# Common Pitfalls

Things that tripped up this build, and things a future contributor is likely to hit.

## Backend

- **AutoMapper version vs. .NET 8**: AutoMapper releases that patch the recursive-mapping DoS advisory (`GHSA-rvv3-g6hj-g44x`, `CVE-2026-32933`) — 15.1.1 and 16.x — pull in `Microsoft.Extensions.Logging.Abstractions` 10.0.0, which conflicts with a .NET 8 target (`NU1605` package downgrade error). This project pins `AutoMapper` to **13.0.1**, which builds cleanly on net8.0 and carries only an informational `NU1903` advisory warning. That warning is currently low-risk because Phase 1 defines zero mapping profiles. **Re-evaluate this pin before adding mapping profiles with deeply nested/self-referencing object graphs** (the actual attack surface the advisory describes), or when upgrading the whole solution to .NET 9/10.
- **`AddIdentity<TUser, TRole>()` doesn't resolve from a plain class library.** It's only available where the ASP.NET Core shared framework (`Microsoft.AspNetCore.App`) is referenced, which `Microsoft.NET.Sdk.Web` projects get implicitly and `Microsoft.NET.Sdk` class libraries don't. That's why Identity registration lives in `HelpDesk.Api`, not `HelpDesk.Infrastructure`, even though the EF Core Identity *stores* (`AddEntityFrameworkStores<AppDbContext>()`, which only needs `Microsoft.AspNetCore.Identity.EntityFrameworkCore`) are fine in Infrastructure.
- **`HealthCheckOptions` vs. `HealthReport` live in different namespaces**: `Microsoft.AspNetCore.Diagnostics.HealthChecks.HealthCheckOptions` (the ASP.NET Core middleware config type) vs. `Microsoft.Extensions.Diagnostics.HealthChecks.HealthReport` (the underlying health-check result type). Mixing them up produces a confusing "type not found" error that looks like a missing package when it's actually a missing `using`.
- **`dotnet ef` design-time commands always throw `HostAbortedException`** when the app uses top-level statements and the Serilog two-stage bootstrap (`try { ... } catch (Exception ex) { Log.Fatal(...) }`). This is expected — the EF tooling intentionally aborts the host after harvesting the service provider, and the exception is what gets logged. Don't chase it as a real startup failure; check for the `Done.` line (or the actual migration/script file) to confirm success.
- **SQL Server Express isn't installed on every machine** (it wasn't on this one). LocalDB (`sqllocaldb`, connection string `(localdb)\MSSQLLocalDB`) is the standard substitute for local development and is what this project's `appsettings.Development.json` targets. Swap the connection string for a real SQL Server/Express instance when one is available; nothing else needs to change.
- **User secrets are per-machine.** The `Jwt:SecretKey` used by `HelpDesk.Api` is stored via `dotnet user-secrets`, not in `appsettings.*.json`. A fresh clone needs to run `dotnet user-secrets set "Jwt:SecretKey" "<some-32+-char-value>"` from `backend/src/HelpDesk.Api` before the app can validate JWTs.
- **`SignInManager<T>` doesn't resolve from `HelpDesk.Infrastructure`, but `UserManager<T>`/`RoleManager<T>` do.** A quick spike (add the type to a throwaway class, try to build) settled this precisely instead of guessing: `SignInManager` needs `IHttpContextAccessor`/`IAuthenticationSchemeProvider` from the ASP.NET Core web framework, same constraint as `AddIdentity` above. It doesn't matter here — a stateless JWT API authenticates via `UserManager.CheckPasswordAsync` directly and has no use for `SignInManager`'s cookie-based sign-in, so `AuthService` lives correctly in `Infrastructure`.
- **`dotnet run --no-launch-profile` does not use the port from `launchSettings.json`.** It falls back to Kestrel's own default (`http://localhost:5000`), not the `5019` the "http" profile defines. If you need a specific port without the launch profile, set `ASPNETCORE_URLS` explicitly.
- **Claims added via `new JwtSecurityToken(issuer, audience, claims, ...)` are NOT short-name-remapped.** `JwtSecurityTokenHandler`'s outbound short-name mapping (`ClaimTypes.NameIdentifier` → `"nameid"`, etc.) only applies when a token is built from a `ClaimsIdentity`/`SecurityTokenDescriptor` via `CreateToken`/`WriteToken`. Constructing a `JwtSecurityToken` directly from a `List<Claim>` (as `TokenService` does) preserves the full `ClaimTypes.*` URI as-is in the token payload. This is harmless for validation (the inbound handler just passes already-long claim types through unchanged, so `ICurrentUserService` still reads them correctly) — but a test asserting on the JWT's raw claim `Type` string must check against `ClaimTypes.NameIdentifier`, not the short JWT name `"nameid"`.
- **Refresh tokens are stored as plaintext in the database**, not hashed. Acceptable for this stage since the table isn't otherwise exposed, but hash them (HMAC + pepper, compare on lookup) before treating this as production-ready.
- **401 vs. 403 is an easy mix-up in a custom exception hierarchy.** `UnauthorizedAppException` was originally used both for "you're not logged in / bad credentials" (genuinely 401) and "you're logged in but this isn't your resource" (should be 403). Live-testing "Employee views someone else's ticket" surfaced this as an incorrect 401. Split it: added `ForbiddenAppException` (403) for resource-ownership/permission checks, keeping `UnauthorizedAppException` strictly for authentication failures. If you add more `AppException` subtypes, ask "is the caller unknown, or just not allowed here?" — those are different status codes.
- **`IQueryable<Category>` assigning to `IQueryable<LookupEntity>` (the base class) works fine** — `IQueryable<T>` is covariant (`IQueryable<out T>`), and EF Core's query provider still translates `Where`/`OrderBy`/`Select` correctly against the upcast handle, since the runtime provider is unchanged. Useful for a shared "get ordered lookup items" helper (`LookupService`) instead of three near-identical methods.
- **A ticket-history "diff" needs the *old* values captured before you mutate the tracked entity.** `TicketService.UpdateAsync` reads `ticket.Title`/`ticket.CategoryId`/etc. into the `RecordHistory` call *before* assigning the new value to the same property — reversing that order silently records "changed from X to X" for every field.
- **Round-robin rotation state doesn't need its own singleton row.** The first instinct was a small `RoundRobinState { LastAssignedAgentId }` table, upserted on every auto-assignment. Simpler and equally correct: derive "who was assigned last" by querying the most recent `TicketAssignments` row where `AssignmentType == RoundRobin`, ordered by `CreatedAt` descending. The assignment-history table *is* the state — a separate counter would just be a second source of truth that could drift from it.
- **A notification triggered by user-supplied content (an @mention) must re-check authorization at send time, not just at write time.** A comment's `IsInternal` flag already gates who can *read* it, but a naive implementation would still fire a mention notification to anyone named in the text — including, for an internal note, the Employee who filed the ticket. That leaks the existence (and title) of an internal note through a side channel even though the Employee can never open the comment itself. `CommentService` re-checks the mentioned user's role before notifying when the comment is internal.
- **When three services all need the same "can this user see this ticket" check, extract it before the third copy, not after.** `TicketService` (Phase 3) had a private `GetTicketForCurrentUserAsync`. Phase 4 added `AssignmentService` and `CommentService`, both needing the identical rule. Copy-pasting a third private method would have meant three places to update if the rule ever changed (e.g. adding a per-category permission later); moved to a shared static `TicketAccessGuard` in `Infrastructure/Services/Shared/` instead, and refactored `TicketService` to use it too so there's exactly one implementation.
- **Removing a field from a request DTO is a real breaking change to trace through, not just a delete.** Moving ticket assignment out of `UpdateTicketRequest` (Phase 3) into its own `AssignTicketRequest` (Phase 4) meant updating: the validator, the service method, existing unit tests that constructed the old DTO shape, and the frontend's `UpdateTicketInput` type, its edit-form payload builder, and the "assign to me" UI that used to live on the edit form. Grep for the removed property name across both `backend/` and `frontend/` before considering the removal done.
- **`AppDbContext.SaveChanges` always stamps `CreatedAt = UtcNow` on insert (see `ApplyAuditInformation`), which breaks tests that need historical `CreatedAt` values** (e.g. dashboard tests seeding tickets across several calendar months). The fix isn't to special-case the override — it's a two-step insert in the test itself: save the entity once (so it's tracked and `CreatedAt` gets the real-but-irrelevant "now" stamp), then set the desired historical `CreatedAt` and save again. On the second save the entry is `Modified`, not `Added`, and `ApplyAuditInformation` only touches `UpdatedAt` for `Modified` entries — the manually-set `CreatedAt` survives.
- **QuestPDF throws at runtime unless a license is selected**, via a global static (`QuestPDF.Settings.License = LicenseType.Community`). Setting it once in `Program.cs` covers the running app but **not** the test host, since xUnit never executes `Program.cs`'s top-level statements — any test that actually calls `GeneratePdf()` needs its own `QuestPDF.Settings.License = LicenseType.Community` (a static constructor in the test class works well, since the setting is idempotent).
- **SLA compliance needs a target to measure against, and this project doesn't have a configurable SLA-policy table.** Rather than invent one (a real admin-phase feature), Phase 5's SLA report reuses `Ticket.DueDate` — a field that already existed since Phase 1 — as the target deadline. This is a deliberate scope decision, not an oversight: a ticket with no due date is simply excluded from SLA tracking rather than falling back to some invented default policy.
- **A relative file-storage path needs to be resolved against the same base directory the rest of the app already uses for relative paths, not `AppContext.BaseDirectory` (the compiled bin output).** `LocalFileStorageService`'s `FileStorage:RootPath` config resolves against `Directory.GetCurrentDirectory()` — the same convention Serilog's `Logs/log-.txt` path already relies on in this project for `dotnet run` — specifically so both land relative to the same content root instead of one ending up several directories deeper inside `bin/Debug/net8.0/`.
- **`IList<string>` (what `UserManager.GetRolesAsync` returns) does not implicitly convert to `IReadOnlyList<string>`**, despite `IReadOnlyList` conceptually being the weaker interface — .NET's `IList<T>` doesn't inherit `IReadOnlyList<T>`. A plain `.ToList()` on the result fixes the mismatch; don't assume "more restrictive" collection interfaces are always freely assignable from "less restrictive" ones in .NET.
- **A generic EF Core service (`AdminLookupService<TEntity> where TEntity : LookupEntity, new()`) is the right call when several entities are truly identical in shape** (here: `Category`/`Priority`/`Status` — all just `Name`/`Description`/`DisplayOrder`/`IsActive`). One generic `Infrastructure` service backed by `_dbContext.Set<TEntity>()`, registered as an open generic (`services.AddScoped(typeof(IAdminLookupService<>), typeof(AdminLookupService<>))`), replaced what would otherwise have been three near-duplicate CRUD services. The controller layer still needs one thin action group per entity (ASP.NET routing doesn't generalize as cleanly), but that's unavoidable framework ceremony, not logic duplication.

## Frontend

- **Never run two `npm install` invocations concurrently in the same project.** Doing so during this build corrupted `node_modules` (`ENOTEMPTY`/`EPERM` errors on partially-extracted packages). The fix was deleting `node_modules` and `package-lock.json` and reinstalling once, sequentially. npm's install lock does not fully protect against this on Windows.
- **TypeScript 6's `baseUrl` is deprecated** (`TS5101`). Path aliases (`@/*` → `./src/*`) work fine via `paths` alone once `baseUrl` is removed — don't add it back when copying tsconfig snippets from older tutorials.
- **shadcn/ui's CLI prompts interactively even with `-y`** in non-TTY shells (it still asks "Select a component library" / "Which preset?"). Pipe answers with `yes "" | npx shadcn@latest init -t vite -y` (or run it from an actual interactive terminal) rather than assuming `-y` alone is silent.
- **The Base UI-flavored shadcn `Button` doesn't support Radix's `asChild` prop.** It's a different primitives library (`@base-ui/react/button`) with its own polymorphism convention. Rendering a `Button` as a `Link` should go through `buttonVariants({ variant })` applied directly to the `<Link>`'s `className`, not `<Button asChild><Link/></Button>`.
- **The default Vite template ships `oxlint`, not ESLint.** If your CI/review process expects "no ESLint errors," you need to add ESLint yourself (`eslint`, `@eslint/js`, `typescript-eslint`, `eslint-plugin-react-hooks`, `eslint-plugin-react-refresh`, `globals`) and a flat `eslint.config.js` — oxlint alone won't satisfy that check.
- **A `QueryClient` (or any object with mutable state) must not be created inline in JSX/render body.** It's constructed once via `useState(() => new QueryClient(...))` in `QueryProvider` so it survives re-renders; creating it as a plain `const` at module scope would work too, but creating it directly in the component body would re-instantiate it on every render.
- **A naive Axios 401-refresh interceptor causes a refresh storm** if several requests 401 at once (e.g. on a page that fires multiple queries in parallel after an access token expires) — each would otherwise trigger its own `POST /auth/refresh-token` call, racing to rotate the same refresh token and invalidating each other. `axiosInstance.ts` shares one in-flight `refreshPromise` across all callers so concurrent 401s trigger exactly one refresh call.
- **The refresh interceptor must exclude the login and refresh-token endpoints themselves** from triggering another refresh attempt on a 401 (checked via `originalRequest.url`), otherwise a wrong-password login response would incorrectly try to "refresh" and loop.
- **@mentions were built as a plain-text token (`@[Name](id)`), not a rich-text editor.** A contentEditable-based mention plugin would need its own serialization format and a heavier dependency; storing the mention as inline markdown-like text keeps the comment `content` a plain string end-to-end (matches the existing `TicketComment.Content` column, no schema change) and lets the frontend recover both the display name and the ID with one regex pass for rendering (`MentionText`) and notification targeting (backend `MentionParser`) without them needing to agree on anything beyond the token shape.
- **`useEffect` + `setState` to sync a form with async-loaded data trips `eslint-plugin-react-hooks`' `set-state-in-effect` rule**, and rightly so — it's a cascading-render pattern. `SystemSettingsPage` first tried `useEffect(() => { if (settings) setForm(settings) }, [settings])`; the lint error caught it before it shipped. The fix is the same one `EditTicketPage` already established in Phase 3: pass the query data straight into React Hook Form's `values` option (`useForm({ values: settings })`) instead of mirroring it into separate local state.

## General

- Background shell commands that end with an unbacked `&` (e.g. `dotnet run &` inside a single `Bash` call with `run_in_background: true`) can report "completed" as soon as the *wrapper* shell exits, even though the backgrounded child process is still running or has been silently killed — always verify with an actual request (`curl`) rather than trusting the "completed" status alone for long-running servers.
