# Future Development Phases

Per `PROJECT_SPEC.md`: each phase must be fully completed, tested, and documented before the next begins.

- **Phase 1 — Foundation** ✅ — Clean Architecture solution, all entities, EF Core migration + seed data, Swagger/health/versioning/logging/CORS/JWT-config/Identity-config, frontend scaffold with routing/layouts/placeholder pages, baseline tests.
- **Phase 2 — Authentication** ✅ — Register, login, logout, refresh tokens, forgot/reset/change password, profile update, JWT issuance, role- and policy-based authorization, activity logging; frontend auth context, `useAuth`, real `ProtectedRoute` guard, Axios refresh-token interceptor, persistent login. See `PHASE2_AUTHENTICATION.md`.
- **Phase 3 — Ticket Management** ✅ — Create/Get/Update/Soft Delete/Restore, search/filter/sort/pagination, automatic ticket history; frontend Ticket List/Details/Create/Edit with RHF+Zod+TanStack Query. See `PHASE3_TICKET_MANAGEMENT.md`.
- **Phase 4 — Ticket Workflow** ✅ — Manual/round-robin assignment with dedicated history, public comments and Agent+-only internal notes, @mentions, in-app notifications (replacing `NoOpNotificationService`) with an email stub; frontend Assignment Panel, Comments/Internal Notes, mention autocomplete + highlighting, Notification Center, unified ticket timeline. See `PHASE4_TICKET_WORKFLOW.md`.
- **Phase 5 — Dashboards & Reporting** ✅ — KPI summary, category/priority breakdowns, monthly trend, resolution time, SLA report (measured against each ticket's own due date); PDF (QuestPDF) and Excel (ClosedXML) export of the same data; frontend Dashboard (KPI cards, pie/line/bar charts, SLA section) and Reports page. See `PHASE5_DASHBOARDS_REPORTING.md`.
- **Phase 6 — Administration** ✅ — Secure ticket file upload/download (storage outside web root, extension/size validation), Admin APIs for Users/Roles/Categories/Priorities/Statuses/Activity Logs/System Settings; frontend Admin Dashboard, User/Category/Priority/Status Management, System Settings, Activity Log Viewer, Attachments Panel. See `PHASE6_ADMINISTRATION.md`.
- **Phase 7 — Hardening** ✅ — Security response headers + HSTS, fail-fast JWT secret-length validation, failed-login activity logging, magic-byte upload content validation; 44 new backend tests (unit + real-pipeline integration/API — auth flow, authorization, CORS/SQLi/XSS, ticket CRUD) bringing the suite to 149 passing; frontend `ErrorBoundary`, route-based code splitting, accessibility labeling pass; complete Postman collection, testing report, coverage report. See `PHASE7_HARDENING.md`.
- **Phase 8 — Production Readiness** ✅ *(this phase)* — Dockerfiles for API and frontend (multi-stage, non-root, container healthchecks), `docker-compose.yml` orchestrating API + frontend + SQL Server with persistent volumes and healthcheck-gated startup; Swagger available in every environment + exported `swagger.json`; Setup Guide, Deployment Guide, User Guide, Deployment/Production Checklists, Release Notes, Presentation Summary. Tagged `v1.0.0`. See `PHASE8_PRODUCTION_READINESS.md`.
- **(Later)** — Real SQL Server edition (not Express) sized for production volume, secrets via a managed vault instead of `.env`/`dotnet user-secrets`, TLS termination in front of the stack, hash refresh tokens at rest, real email provider instead of `LoggingEmailSender`, ticket number generation via a DB sequence (race-condition-safe), real-time notification delivery (SignalR/WebSockets) instead of polling, configurable per-priority SLA policies, custom role creation, rate-limiting on login/register, frontend automated test suite (Vitest/Testing Library), CI/CD pipeline. Full detail in `PRODUCTION_CHECKLIST.md`.
