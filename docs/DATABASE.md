# Database Schema

## Overview

The database is `HelpDeskSystemDb`, managed entirely through EF Core migrations against `HelpDesk.Infrastructure.Persistence.AppDbContext`. It combines two sets of tables:

1. **ASP.NET Identity tables** (`AspNetUsers`, `AspNetRoles`, `AspNetUserRoles`, `AspNetUserClaims`, `AspNetRoleClaims`, `AspNetUserLogins`, `AspNetUserTokens`) — generated by `IdentityDbContext<ApplicationUser, ApplicationRole, Guid>`.
2. **Domain tables** — one per entity in `HelpDesk.Domain.Entities`: `Categories`, `Priorities`, `Statuses`, `Tickets`, `TicketComments`, `TicketAttachments`, `TicketHistories`, `Notifications`, `ActivityLogs`, `RefreshTokens` (added in Phase 2).

All primary keys are `Guid` (`uniqueidentifier`), generated client-side by convention (`ValueGeneratedOnAdd`).

## Entity Relationships

```
ApplicationUser (AspNetUsers)
  ├─1:N──▶ Ticket            (CreatedByUserId, restrict delete)
  ├─1:N──▶ Ticket            (AssignedToUserId, nullable, restrict delete)
  ├─1:N──▶ TicketComment     (UserId, restrict delete)
  ├─1:N──▶ TicketAttachment  (UploadedByUserId, restrict delete)
  ├─1:N──▶ TicketHistory     (ChangedByUserId, nullable, restrict delete)
  ├─1:N──▶ Notification      (UserId, cascade delete)
  ├─1:N──▶ ActivityLog       (UserId, nullable, set null on delete)
  └─1:N──▶ RefreshToken      (UserId, cascade delete)

Category  ──1:N──▶ Ticket   (CategoryId, restrict delete)
Priority  ──1:N──▶ Ticket   (PriorityId, restrict delete)
Status    ──1:N──▶ Ticket   (StatusId, restrict delete)

Ticket
  ├─1:N──▶ TicketComment     (cascade delete)
  ├─1:N──▶ TicketAttachment  (cascade delete)
  ├─1:N──▶ TicketHistory     (cascade delete)
  └─1:N──▶ Notification      (RelatedTicketId, nullable, restrict delete)
```

**Delete-behavior rationale:** every FK pointing *to* `AspNetUsers` (except `Notifications` and `ActivityLogs`) is `Restrict`, and every FK from `Ticket`'s child records back to `Ticket` is `Cascade`. This avoids SQL Server's "multiple cascade paths" error — if both `Ticket → TicketComment` and `ApplicationUser → TicketComment` cascaded, deleting a ticket and a user in the same operation would create an ambiguous cascade path. Restricting the user-side FKs sidesteps that entirely. `ActivityLogs.UserId` uses `SetNull` since an activity log entry should outlive the user who caused it.

## Soft Delete

Every entity deriving from `BaseEntity` (all domain entities) and `ApplicationUser` implements `ISoftDelete` (`IsDeleted`, `DeletedAt`, `DeletedBy`). `AppDbContext`:
- Applies a global EF Core query filter (`HasQueryFilter`) on every `ISoftDelete` entity, so `IsDeleted == true` rows are excluded from all default queries (use `.IgnoreQueryFilters()` to see them).
- Intercepts `SaveChanges`/`SaveChangesAsync`: any entity tracked as `Deleted` is converted to `Modified` with `IsDeleted = true` and `DeletedAt` stamped — so calling `dbContext.Remove(entity)` never issues a `DELETE`, only an `UPDATE`.

## Auditing

`IAuditableEntity` (`CreatedAt`, `CreatedBy`, `UpdatedAt`, `UpdatedBy`) is stamped automatically in the same `SaveChanges` override: `CreatedAt` on `Added`, `UpdatedAt` on `Modified`. `CreatedBy`/`UpdatedBy` remain unpopulated for now — a future phase can set them from `ICurrentUserService`, which is fully wired since Phase 2 (backed by real JWT claims).

## Refresh Tokens (Phase 2)

`RefreshTokens` stores one row per issued refresh token: `Token` (unique, plaintext — see `PITFALLS.md` for the hash-at-rest trade-off), `ExpiresAt`, and rotation metadata (`RevokedAt`, `RevokedByIp`, `ReplacedByToken`, `CreatedByIp`). Tokens rotate on every refresh (the old row is revoked and linked via `ReplacedByToken` to the new one) rather than being reused, so a captured-and-replayed old token is rejected. `IsActive`/`IsExpired`/`IsRevoked` are computed properties (`[NotMapped]`-equivalent via `builder.Ignore(...)` in `RefreshTokenConfiguration`), not columns.

## Indexes

- Every `BaseEntity` table has an index on `IsDeleted` (soft-delete filtering).
- `Categories.Name`, `Priorities.Name`, `Statuses.Name`, and `Tickets.TicketNumber` are unique indexes.
- `Tickets` additionally indexes `StatusId`, `PriorityId`, `CategoryId`, `AssignedToUserId`, `CreatedByUserId`, and `CreatedAt` — the columns a ticket list/filter view will query by.
- Child tables (`TicketComments`, `TicketAttachments`, `TicketHistories`, `Notifications`) index their `TicketId`/`UserId` foreign keys.

## Seed Data

Seeded via `HasData` in each entity's `IEntityTypeConfiguration<T>`, using fixed GUIDs (`Persistence/Seed/SeedIds.cs`) so the seed is idempotent and deterministic across environments:

| Table | Seeded rows |
|---|---|
| `AspNetRoles` | Admin, IT Support Agent, Employee, Manager |
| `Categories` | Hardware, Software, Network, Email, Access Request, Other |
| `Priorities` | Low, Medium, High, Critical |
| `Statuses` | Open, In Progress, Pending, Resolved, Closed |

## Migrations & SQL Script

- `InitialCreate` (`backend/src/HelpDesk.Infrastructure/Persistence/Migrations/20260706172924_InitialCreate.cs`) — Phase 1: all tables except `RefreshTokens`
- `AddRefreshTokens` (`backend/src/HelpDesk.Infrastructure/Persistence/Migrations/20260706182946_AddRefreshTokens.cs`) — Phase 2: the `RefreshTokens` table
- Idempotent SQL script (both migrations): `backend/database/InitialCreate.sql` (regenerated via `dotnet ef migrations script --idempotent`, so the filename now covers the full schema through Phase 2 despite the name)
- Applied to: SQL Server LocalDB, `(localdb)\MSSQLLocalDB`, database `HelpDeskSystemDb`

To regenerate or apply elsewhere:
```
dotnet ef database update --project backend/src/HelpDesk.Infrastructure --startup-project backend/src/HelpDesk.Api
dotnet ef migrations script --project backend/src/HelpDesk.Infrastructure --startup-project backend/src/HelpDesk.Api --idempotent --output backend/database/InitialCreate.sql
```
